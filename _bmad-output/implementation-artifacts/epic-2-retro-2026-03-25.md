## What Went Well

### 1. Epic 2 delivered the full document loop, not isolated features

The team completed a coherent user workflow: upload, processing visibility, extraction, cabinet management, retry/recovery, and flagging. This matters because Epic 3 now has real processed data and stable document states to build on rather than mocks or placeholders.

### 2. Quality discipline stayed high through substantial backend and frontend scope

By the end of the epic, the repo reached 155 backend tests and 90 frontend tests. The team kept adding targeted regression coverage as each review round exposed gaps, especially around SSE flow, retry semantics, ownership checks, cache invalidation, and accessibility.

### 3. Review feedback was actually applied, not merely acknowledged

Epic 2 shows repeated evidence of findings being closed with real fixes:
- Story 2.1 fixed pre-existing `npm run check` friction and hardened upload semantics.
- Story 2.2 hardened SSE timeout, heartbeat, reconnect, cleanup, and terminal-state handling.
- Story 2.4 and 2.5 revisited deletion/retry consistency several times until the behavior matched the intended contracts.
- Story 2.6 closed the loop with accessibility and concurrency follow-ups instead of deferring them.

### 4. Forward compatibility improved during execution

The team did not just ship current acceptance criteria. It also laid groundwork for the next steps:
- pending retry metadata avoids misattributing old values to replacement files
- value flagging now persists state needed by the future admin correction queue
- document/detail query patterns and cabinet behavior are now explicit enough for Epic 3 dashboard work to reuse

## What Didn't Go Well

### 1. Review churn stayed expensive, especially around consistency contracts

The same class of problems kept resurfacing:
- state consistency during retries and deletes
- cache invalidation holes in document/detail flows
- ownership and defense-in-depth checks in repository/service boundaries
- frontend terminal-state handling for partial/failed paths

The architecture stayed intact, but the cost of reaching the correct behavior remained high.

### 2. "Done" often meant "done after another review round"

Several stories moved forward, then reopened when deeper behavioral bugs were found:
- Story 2.4 oscillated around delete atomicity and MinIO cleanup semantics
- Story 2.5 had to revisit retry metadata authority, failure-state preservation, and immediate UI dismissal
- Story 2.6 needed accessibility and state-sync fixes after initial completion

That pattern suggests the team is still relying on review to discover contract mismatches that should be caught earlier.

### 3. Cross-story invariants are still under-documented

Epic 2 repeatedly depended on implicit rules rather than a stable checklist:
- when document metadata becomes authoritative
- how query invalidation must work for list + detail + derived health-value views
- what "partial" means versus "failed"
- where storage/object cleanup is best-effort versus guaranteed

These rules exist in the code and story notes, but they are not yet compactly codified for fast reuse.

### 4. Toolchain confidence improved, but not uniformly

Story 2.1 removed one pre-existing `npm run check` blocker, but Story 2.2 still documented unrelated frontend typing/config issues outside that story. The immediate Epic 2 path worked, but the wider toolchain still shows pockets of drift.

## Key Insights & Lessons Learned

1. The team can carry a complex cross-layer epic to completion without architectural drift.
   Router/service/repository boundaries held even while the document pipeline expanded across storage, workers, SSE, retries, and UI state.

2. Behavioral contracts are the real complexity center now.
   The hardest problems were not framework setup. They were state authority, retry semantics, partial-versus-failed meaning, cache invalidation, and cleanup guarantees.

3. Review remains the main quality gate for integration logic.
   That is too late for several of Epic 2's most expensive fixes. The project now needs contract-level checks before review starts.

4. Epic 3 is viable because Epic 2 produced reusable domain primitives.
   Documents, extracted values, partial states, retry semantics, and flagging are now concrete enough for dashboard and baseline work.

5. Query and state contracts need to become first-class documentation.
   The next epic will touch dashboard reads, contextual rendering, and trend derivation. That work will move faster if the document/detail/query invalidation rules are written once and reused.

## Next Epic Preview: Epic 3 — Health Dashboard & Baseline Intelligence

### Epic 3 Goals

Epic 3 turns Epic 2's extracted data into user-facing health intelligence:
- baseline recommendations before any upload
- contextual health-value cards
- biomarker trends across uploads

### Dependencies On Epic 2

- processed documents and extracted `health_values`
- reliable document terminal states (`completed`, `partial`, `failed`)
- document/detail query conventions and invalidation behavior
- confidence and flagging fields already exposed to UI consumers

### Preparation Needed Before Heavy Epic 3 Work

| Need | Why It Matters |
|------|----------------|
| Write down query-key and invalidation conventions for documents/detail/health-values/dashboard | Epic 3 will multiply read surfaces and stale-data risks |
| Document status semantics for `pending` / `processing` / `partial` / `failed` / `completed` | Dashboard and trend UI need consistent treatment of partial/confidence states |
| Confirm baseline/dashboard contracts in API shape before UI expansion | Prevent another cycle of review-driven contract discovery |
| Decide how flagged and low-confidence values should appear in dashboard context views | Epic 3 depends on that presentation rule |

### Readiness Assessment

Epic 3 is ready to start, but it should begin with explicit contract-setting around read models and cache semantics rather than jumping straight into UI rendering.
