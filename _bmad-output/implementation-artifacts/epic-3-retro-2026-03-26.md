## What Went Well

### 1. Epic delivered a coherent user journey, not isolated widgets

Story 3.0 raised the visual floor. Story 3.1 solved cold-start. Story 3.2 added context intelligence. Story 3.3 added temporal understanding. Each story built cleanly on the previous one. Epic 4 inherits real, working UI surfaces with established patterns.

### 2. Test coverage grew consistently and tracked new behavior

From 245 (end of Epic 2) to 168 backend + 109 frontend. Every story added targeted regression coverage rather than relying on luck.

### 3. Pragmatic technical decisions landed well

Story 3.3's raw SVG fallback (zero-dependency) avoided the React-only Recharts trap called out in the architecture doc. The decision to derive sparklines from the existing `values` array rather than N+1 `getHealthValueTimeline()` calls was architecturally sound and performance-conscious.

### 4. TanStack Query migration was forward-looking

Dashboard migration in Story 3.3 means SSE invalidation now covers `['health_values']`, `['baseline']`, and `['timeline']`. Epic 4's AI streaming and real-time update surface inherits a reactive data layer that would have been much harder to add later.

### 5. Empty state consistency held across 3 consecutive stories

Stories 3.1, 3.2, and 3.3 each added significant dashboard complexity without breaking the empty state. The explicit antipattern lists in story notes ("Do NOT refactor the empty state") were effective guardrails.

### 6. Contract documentation in story notes worked

The Epic 2 retro introduced the habit of writing explicit contracts and antipatterns. Stories 3.2 and 3.3 benefited visibly — fewer regressions in areas covered by explicit notes.

## What Didn't Go Well

### 1. Docker Compose frontend test environment remains blocked

Story 3.0 documented: *"Docker Compose frontend test command could not be completed — runtime image excludes source test files."* This was not resolved in 3.1, 3.2, or 3.3. The project rule requires tests to run inside Docker Compose, but they structurally cannot. Every story in Epic 3 is technically non-compliant with the testing rule.

### 2. Review patch count remained high throughout the epic

| Story | Patches | Decisions |
|-------|---------|-----------|
| 3.0 | 3 | 0 |
| 3.1 | 5 | 1 |
| 3.2 | 9 | 1 |
| 3.3 | 10 | 0 |

The trend is upward as stories grew more complex. Review is catching real bugs, but many patches represent implementation errors that shouldn't have reached review.

### 3. Svelte 5 runes patterns still causing implementation errors

Story 3.3's review caught `$derived(...)` used where `$derived.by(...)` was required — a critical reactivity bug where the derived holds a function object instead of a value. This is the second consecutive epic where Svelte 5 rune misuse was found in review. The team is still not internalizing the pattern.

### 4. Architecture doc has diverged from the codebase

The architecture doc references "Recharts (Tailwind-native)" which is React-only and unusable in Svelte 5. It also references files that do not exist (`(app)/dashboard/+page.ts`, `frontend/src/lib/api/health-data.ts`). Story planning consumed real effort compensating for a document that's no longer reliable.

### 5. Pre-existing test failures silently accumulating

Story 3.3 noted "Pre-existing failures (9) in documents/upload tests are unrelated to this story." These were deferred rather than addressed. They are now carried forward into Epic 4.

---

## Key Insights & Lessons Learned

1. **Explicit antipatterns in story notes prevent regressions.** The "Do NOT" lists in 3.2 and 3.3 worked. This habit should be preserved and extended to Epic 4.

2. **Svelte 5 rune patterns need a shared reference.** `$derived` vs `$derived.by`, `$effect` cleanup guards, TanStack Query wrapper patterns — the team is relearning these in every review. A one-page cheat sheet would prevent a recurring class of review patches.

3. **The Docker Compose frontend test issue is infrastructure, not a story gap.** It cannot be resolved by a story note. It requires a targeted fix to the container build.

4. **Review is doing real work.** NaN/Inf guards, ARIA role correctness, zero-span reference range edge cases, sparkline sort ambiguity, reference band using latest vs oldest data point — these were genuine defects. The review process is not overhead; it's catching real bugs. The goal should be moving more of this catch to earlier in the process.

5. **Architecture docs decay and the cost compounds.** The Recharts mistake could have caused real implementation churn. Before Epic 4 — which introduces LangGraph, `safety.py`, embeddings, and streaming — the architecture doc needs to reflect actual repo state.

## Next Epic Preview: Epic 4 — AI Health Interpretation

### Epic 4 Goals

Epic 4 adds AI interpretation to everything Epic 3 built:
- Plain-language AI interpretation per uploaded result (Story 4.1)
- AI reasoning trail with source visibility (Story 4.2)
- Follow-up Q&A grounded in full user history (Story 4.3)
- Cross-upload pattern detection (Story 4.4)

### New Technology Surface

Epic 4 introduces substantially new risk area: Anthropic Claude API, LangGraph pipelines, `safety.py` validation, encrypted `ai_memory` table, Voyage-3 embeddings, and streaming SSE responses. This is a different complexity tier from Epic 3's dashboard/visualization work.

### Dependencies On Epic 3

- TanStack Query reactive layer (dashboard migrated in Story 3.3 — SSE invalidation ready)
- Established `HealthValueBadge`, `BiomarkerTrendChart`, and value card components for Story 4.1 integration
- Dashboard active/empty state branching — AI cards will extend the active state established in 3.2

### Hard Blocker: EU DPA with Anthropic

Story 4.1 AC explicitly requires: *"a signed Data Processing Agreement (EU-compatible DPA) is in place"* and *"no user health data is sent to any AI provider without a DPA."* This is a legal gate, not a technical one. Story 4.1 cannot be implemented without it.

### Preparation Needed Before Epic 4 Stories Are Created

| Need | Why It Matters |
|------|----------------|
| Fix Docker Compose frontend test environment | AI streaming components will need real test infrastructure; carrying this into a more complex epic makes it worse |
| Audit `app/ai/` and `safety.py` current state vs architecture doc | Architecture doc has drifted; verify actual module state before designing AI stories |
| Confirm EU DPA with Anthropic is in place | Hard legal blocker for Story 4.1 |
| Define `ai_memory` schema and encryption contract | Encrypted storage + 1024-dim embeddings require careful upfront design |
