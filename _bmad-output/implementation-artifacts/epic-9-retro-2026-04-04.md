## What Went Well

1. **Velocity flywheel** — Story 9-1 established all patterns (CSS naming, component structure, testing approach). Stories 9-3 and 9-4 were significantly faster because they extended proven patterns. All 4 stories were created, implemented, reviewed, and completed in a single session.

2. **CSS naming convention** — `.hc-[section]-[element]` pattern (e.g., `.hc-app-header`, `.hc-admin-nav-item`, `.hc-toast-dismiss`) provided consistency across 40+ CSS classes. All styles in `app.css` only — no scoped style blocks.

3. **3-layer adversarial code review** — Blind Hunter, Edge Case Hunter, and Acceptance Auditor caught real issues: aria-live timing, dead CSS, stale comments, accessibility violations. Every story passed all acceptance criteria.

4. **Accessibility testing baseline** — Every new component includes axe-core audit in its test file. Test count grew from 341 to 377 with comprehensive coverage.

5. **Design system reuse** — StatusBar, StatusBarField, Button, Panel components from Epic 7 were consistently reused. AdminShell reuses 6 shared CSS classes from AppShell.

## Challenges

1. **Accessibility caught in review, not implementation** — Every story had accessibility issues found during code review:
   - 9-1: Missing skip-to-content link (WCAG 2.4.1), `width: 100vw` horizontal overflow
   - 9-2: No accessibility issues (removal-only story)
   - 9-3: No new accessibility issues (followed 9-1 patterns)
   - 9-4: `aria-live="polite"` on conditionally rendered container, `role` attribute on generic div without landmark role

   **Root cause:** Accessibility knowledge isn't embedded in the implementation phase. Dev agents don't have a WCAG checklist during coding — they rely on review to catch violations.

2. **Spec-vs-reality ambiguity** — In 9-2, the spec said to remove `max-width` constraints. Reviewers flagged this as a visual regression risk. Resolution required a decision point (keep removed per spec, since Epic 8 layout already handles it). This type of ambiguity slows the review cycle.

3. **Deferred debt multiplication** — The `handleSignOut` click test gap was deferred in 9-1 review and carried forward to 9-3 (AdminShell copied the pattern). Deferred items propagate when patterns are reused.

## Key Insights

1. **Pattern investment pays exponential returns** — Story 9-1 was the heaviest lift (full rewrite, establishing conventions). Each subsequent story was faster because it extended proven patterns rather than inventing new ones.

2. **"All CSS in app.css" scales well** — For a design system with ~40+ custom classes, having everything in one file with section comments (`/* ── Admin Shell ── */`, `/* ── Toast Notifications ── */`) is navigable and prevents scoping conflicts.

3. **Svelte 5 runes work well for stores** — The `FeedbackStore` class with `$state` and `$derived` is clean and testable. Module-level singleton pattern (like `feedbackStore`) works for client-only apps with `ssr: false`.

4. **Desktop-only simplifies everything** — Removing responsive code (9-2) was a one-time investment that makes every future story simpler. No `sm:`, `md:`, `lg:` prefixes to think about.

## Action Items

### Process Improvements

1. **Add WCAG guidelines to story creation** — Each story's Dev Notes should include a "WCAG Considerations" section calling out relevant accessibility patterns (aria-live, focus management, roles, contrast).

2. **Shift accessibility checks into implementation** — Dev agent should run axe audit after each task, not just at the end. Catch violations earlier in the cycle.

### Team Agreements

- All new components get axe accessibility audits in their test files
- CSS stays in `app.css` only, no scoped `<style>` blocks
- `.hc-[section]-[element]` naming convention for all new CSS classes
- Error toasts persist until dismissed; success/warning auto-dismiss at 3s
- Desktop-only: no responsive prefixes (enforced by 9-2)
