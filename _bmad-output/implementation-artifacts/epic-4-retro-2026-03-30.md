## What Went Well

**1. Core AI architecture is solid and consistent**
The `inject_disclaimer → validate_no_diagnostic → surface_uncertainty` safety pipeline, established in 4-1, was applied across all stories with intentional and documented variations. Non-fatal error handling meant the app never broke when AI calls failed.

**2. Encryption uniformly applied**
Every AI content store (interpretation, reasoning context) went through the encryption layer. No story introduced plaintext AI content in the DB.

**3. Story 4-3 demonstrated the right approach**
Four review patches. Clean streaming implementation. The difference: dev notes locked down endpoint signatures, return types, and architecture decisions *before* implementation started. This is the template for all future stories.

**4. DPA blocker from Epic 3 resolved before Epic 4 began**
This could have stopped the entire epic. It didn't.

**5. Svelte 5 cheat sheet paid off immediately**
Zero Svelte rune errors flagged in 4-3 or 4-4 reviews. Compare to repeated rune errors across Epic 3.

**6. deferred-work.md is an honest, precise ledger**
Every known issue documented with exact file:line references. Unusual discipline — most teams lose tech debt to tribal knowledge.

**7. Full product vision delivered**
Users now have: AI interpretation, reasoning trail transparency, follow-up Q&A, and cross-upload pattern detection. Complete Epic 4 scope shipped.

---

## What Didn't Go Well

**1. Architecture decisions made during code review, not before implementation**
High patch counts on 4-1, 4-2, 4-4 share a root cause: endpoint paths, return type signatures, and pipeline variations were discovered or corrected in review. Story 4-3 broke this pattern — its dev notes were precise. The others were not.

**2. Story 4-4 required two full review rounds**
JSON parsing failures from real Claude output were not anticipated. LLM output is never guaranteed clean — this should have been handled in the initial implementation. The second round was also needed because safety pipeline variations for patterns weren't documented upfront.

**3. Technical debt accumulating faster than it's being resolved**
HIGH-severity items from Epic 1 remain open (worker-delete race, no refresh token revocation). From Epic 4: TOCTOU race in `upsert_ai_interpretation`, safety rejection with no recovery path, Anthropic client initialised at module import breaking test isolation. The ledger is honest but there's no resolution cadence.

**4. Test environment state was unknown for the entire epic**
We could not confirm whether Docker Compose frontend tests were passing. Pre-existing test failure count unknown. If test state is unknown, we're shipping blind.

**5. Mocked AI responses masked real failure modes**
The 4-4 JSON parsing issue (malformed Claude output) would have been caught earlier with real API calls in tests. Simulated/mocked LLM responses create false confidence.

**6. Three Epic 3 action items not addressed**
Docker Compose test environment, pre-existing failures, and architecture doc all carried over unresolved.

## Strategic Decision: Migrate AI Module to LangChain

**Decision:** Migrate the AI module from the Anthropic SDK to LangChain before Epic 5 begins.

**Rationale:** Model portability. The current implementation is coupled to Anthropic/Claude. LangChain provides a model-agnostic abstraction that allows switching to any LLM (GPT-4, Gemini, open-source models) without code changes. This is a strategic principle, not a preference.

**Scope:**
- Replace `claude_client.py` with a LangChain abstraction
- Update `service.py` AI calls to use LangChain chat interface
- Update streaming implementation to use LangChain streaming
- Update all AI-related tests
- Safety pipeline (`inject_disclaimer`, `validate_no_diagnostic`, `surface_uncertainty`) is custom logic — untouched

**Implementation:** Dedicated Story 5.0 — must complete before any Epic 5 story begins.

## Process Principles — Adopted for Epic 5+

**P1: No story closes without confirmed passing tests in Docker Compose.**
Test state must be known, not assumed. Unknown = failing.

**P2: No mocking AI responses in tests. Always test with real API calls.**
Simulated responses mask real failure modes. Integration tests for AI behavior hit the real API and assert on real output. Unit tests for pure logic (non-AI) may remain isolated.

**P3: Endpoint contracts, return types, and pipeline variations locked in dev notes before implementation starts.**
Story 4-3 is the template. If it's not in the dev notes before the first line of code, it will be found in code review — at higher cost.
