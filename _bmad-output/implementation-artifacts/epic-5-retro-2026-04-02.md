## What Went Well

### 1. P3 discipline became normal practice

Epic 4's strongest lesson was applied well. Stories entered implementation with much more explicit contracts, scope boundaries, and repo-specific constraints. That reduced ambiguity and made reviews more about edge cases than basic architecture decisions.

### 2. Reuse across admin stories was strong

`5.2` created the real admin backbone and `5.3` deliberately built on it instead of forking a parallel subsystem. That improved coherence across backend module structure, frontend route shape, and correction-flow reuse.

### 3. Privacy boundaries stayed explicit

The admin epic never lost sight of its main risk: admin workflows must not expose user health data beyond the narrowly required correction and flag-review surfaces. This boundary shows up repeatedly in story design and implementation notes, which is the right kind of repetition.

### 4. The team handled two high-risk platform changes without breaking public contracts

`5.0` changed the AI adapter boundary. `5.4` changed processing orchestration. Both were invasive backend shifts, yet both were framed around preserving caller-visible behavior. That is mature execution.

### 5. Testing discipline improved meaningfully

Epic 5 stories consistently referenced Docker-backed verification, full-regression runs, or both. The project context rule that local green is not enough was actually treated as an operating rule rather than a suggestion.

---

## What Did Not Go Well

### 1. Planning artifacts were still not canonical

Epic 5 required two execution-time addendum stories because planning and architecture were not aligned with the real codebase. `5.0` exists because the retrospective identified a prerequisite that `epics.md` did not contain. `5.4` exists because architecture described a LangGraph processing shape that the repo did not yet implement. This is the biggest systems issue from the epic.

### 2. Review load stayed high on structural stories

The epic still depended on multi-round review to catch boundary issues, hidden state problems, and edge-case regressions. `5.0` and `5.4` in particular needed repeated review passes. The team is improving, but big internal refactors are still discovering too much late.

### 3. Architecture-document drift carried forward again

Epic 5 stories still had to work around stale assumptions such as `python-jose` versus PyJWT, future-state processing paths that were not in the repo yet, and endpoint naming drift in surrounding artifacts. This was an Epic 4 carry-over that remained open after Epic 5 execution.

### 4. Test-environment workflow remained harder than it should be

The team did the right thing by validating in Docker, but multiple stories had to explain that the live `backend` service was not enough and that `backend-test` needed a rebuild for trustworthy results. That is acceptable as a temporary constraint, but it is still operational friction.

### 5. Story artifact hygiene made retrospective analysis harder

Some story files mix resolved and unresolved review notes in ways that are not easy to audit quickly. The implementation may be correct, but the paper trail is not consistently crisp. That increases retrospective effort and weakens tracker trust.

## Significant Discovery Alert

**Epic Update Required: YES**

Epic 5 uncovered a planning conflict that should be resolved before Epic 6 starts:

### Conflict 1: GDPR deletion vs audit-log retention

Epic 6 currently says account deletion removes `audit_logs` rows for the user. Story `5.2` established an opposing implementation assumption: correction audit records survive underlying data deletion and use `ON DELETE SET NULL` to preserve the compliance trail. That is not a minor detail. Epic 6 needs a single explicit policy before any delete flow is built.

### Conflict 2: Epic 6 data inventory is now larger than the original plan

Epic 5 added or operationalized fields and records that Epic 6 must account for:

- `users.account_status`
- `users.last_login_at`
- `health_values.flag_reviewed_at`
- `health_values.flag_reviewed_by_admin_id`
- admin correction audit records

Epic 6 export and delete behavior must define how each of these is handled.

### Conflict 3: Architecture references still contain stale implementation assumptions

Epic 5 story files still had to compensate for:

- `python-jose` references while the project uses PyJWT
- future-state processing structure that only became partially true in `5.4`
- older LangGraph/version assumptions
- stale endpoint naming in adjacent UX/planning material

Epic 6 is a high-compliance epic. Starting it on top of stale architecture text is unnecessary risk.
