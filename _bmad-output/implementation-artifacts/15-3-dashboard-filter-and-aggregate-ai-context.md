# Story 15.3: Dashboard Filter and Aggregate AI Context

Status: done

## Story

As a dashboard user,
I want the dashboard to operate on a persisted filter over my documents and drive AI interpretation and chat from the filtered dataset instead of the latest uploaded document,
So that the dashboard's biomarkers, AI note, and chat all reflect the actual active scope (all docs / analyses only / plain documents only) rather than whatever I happened to upload last.

## Acceptance Criteria

1. **Dashboard filter state** — The dashboard UI exposes a persisted filter with exact values:
   - `all`
   - `analysis`
   - `document`
   Filter state persists across reloads (localStorage). Default on first visit is `analysis` so post-Story-15.2 users immediately see an analysis-scoped dashboard. Filter UI is a 98.css control (radio/select) placed in the dashboard action bar next to "Import Document". The chosen value is the single source of truth for biomarker data, AI interpretation, and AI chat on the dashboard.

2. **Data loaders honor the filter** — Dashboard data fetching respects the active filter:
   - `GET /api/v1/health-values?document_kind={all|analysis|document}` filters values whose owning document has a matching `document_kind`.
   - `document_kind=unknown` documents are NEVER included in any filter outcome on the dashboard (they represent failed/unreadable classification and must not pollute biomarker totals).
   - `document_kind=all` returns values for `analysis`-kind AND `document`-kind documents, still excluding `unknown`.
   - Query param is optional; when omitted, the endpoint preserves the pre-15.3 contract (returns all values including `unknown`). The dashboard always sends the param explicitly.
   - `PatientSummaryBar`'s "Documents" and "Biomarkers" counts are derived from the same filtered dataset — not from the full document count.

3. **Dashboard-scoped AI endpoints** — Two new dashboard-scoped endpoints are added alongside (not replacing) the existing document-scoped `/ai/documents/{id}/interpretation` and `/ai/chat`:
   - `GET /api/v1/ai/dashboard/interpretation?document_kind={all|analysis|document}` — returns an aggregate interpretation built from all of the user's safety-validated `AiMemory` rows whose owning document matches the filter. Response shape mirrors `AiInterpretationResponse` but with `document_id=null` and a new optional `source_document_ids: string[]` listing which documents fed the aggregate.
   - `POST /api/v1/ai/dashboard/chat` with body `{ document_kind: 'all'|'analysis'|'document', question: string }` — streams a follow-up answer using filtered `AiMemory` rows as context. Reuses `stream_follow_up_answer`'s validation/safety pipeline; `active_document_id` is `None` for the dashboard-scope context ordering.
   - Both endpoints enforce ownership via `get_current_user`. Both reject `document_kind=unknown` at the schema layer.
   - When the filter yields zero contributing documents, the interpretation endpoint returns **HTTP 409** with `detail="No analyses available for the active filter"` (same shape as the existing `NoAiContextError` 409 on `/ai/chat`); the chat endpoint returns the same 409 before streaming.

4. **Context rebuilds from persisted data after mutations** — Dashboard AI reflects the current persisted `AiMemory` set after every mutation that changes it:
   - **Delete document**: `AiMemory` cascade-deletes with the `documents` row (already wired); the next dashboard AI request automatically excludes it. No new invalidation call needed, BUT the dashboard `['ai_dashboard_interpretation', kind]` TanStack query MUST be invalidated in the delete success path so the UI re-fetches.
   - **Upload (new document)**: after processing completes, the document's per-document `AiMemory` row is already generated by Story 4.1's `generate_interpretation`. Dashboard AI re-reads this on the next request; the dashboard query is invalidated when the upload's document row transitions to `completed`/`partial` (same hook that already invalidates `['health_values']`).
   - **Reupload**: same as upload — the replacement document's per-document interpretation is regenerated by the existing processing pipeline, and the dashboard query is invalidated on terminal status.
   - **Year confirmation** (`POST /documents/{id}/confirm-date-year` from Story 15.2): the per-document AI interpretation is already invalidated+regenerated in `app/documents/service.py::confirm_date_year`. On the frontend, the dashboard query MUST be invalidated alongside the existing document detail invalidation. No backend change to `confirm_date_year` is required.
   - Because dashboard AI is computed on-demand from the union of per-document `AiMemory` rows, there is no separate dashboard-scope memory table to invalidate. "Rebuild" means "recompute next request from the current row set". Do NOT introduce a second AI persistence path.

5. **Explicit no-analysis states** — When the active filter yields zero contributing documents:
   - `BiomarkerTable` is replaced by an explicit state: `"No {filter-label} yet — upload a lab result to populate this view"` where filter-label is derived (e.g. "analyses", "plain documents", "documents"). The empty state uses the existing 98.css empty-state styling pattern from the dashboard empty state, but is scoped to the filter, not the whole account.
   - `AIClinicalNote` shows a filter-specific empty state (not the generic "AI interpretation is being generated…"): `"No AI interpretation for this filter yet"` with no spinner and no 404 treatment.
   - `AIChatWindow` disables the Send button and shows an inline hint `"Chat is available once this filter has at least one interpreted document"`. The input remains editable so the user can see/draft a question; only submission is blocked.
   - These three empty states are distinct from the pre-15.3 "zero total documents" empty state that greets brand-new users; the filter-empty states are for users who HAVE documents but none match the active filter.

## Dev Notes

### Story Scope and Boundaries

- This is the FIRST story where the dashboard stops depending on `latestDocumentId`. After 15.3, `latestDocumentId` should not appear anywhere in the dashboard page or its components.
- Document-detail AI (`routes/(app)/documents/[id]/+page.svelte`) remains document-scoped. Do NOT change its behavior. The `AIClinicalNote` / `AIChatWindow` dual-mode refactor is load-bearing precisely so the document-detail page can keep calling the existing per-document endpoints unchanged.
- Do NOT merge the document-scoped and dashboard-scoped endpoints. They are kept as parallel routes so the document-detail page is never coupled to a user-wide aggregate.
- Do NOT introduce a second `AiMemory` persistence path (no `ai_dashboard_memory` table, no NULL-`document_id` rows). Dashboard aggregates are computed on-demand from the existing per-document rows. This is load-bearing for AC 4: "rebuild from remaining persisted data" means recomputing next request from the current `AiMemory` set. Anything else creates an invalidation coherence problem.
- Do NOT change the upload or reupload contract. Story 15.4 handles the batch-upload queue.
- Do NOT add localization strings here. Story 15.6 handles EN/UA.
- Multi-file upload is Story 15.4 — this story must keep working before AND after 15.4 lands. Design the filter + dashboard endpoints so that a post-15.4 batch upload (adding N `AiMemory` rows at once) needs no further dashboard-side change, only the existing cache invalidation on terminal SSE status.

- `lib/api/health-values.ts::getHealthValues()` takes no args and hits `/api/v1/health-values` with no query params.

- `lib/types/api.ts:31` already defines `DocumentKind = 'analysis'|'document'|'unknown'` (from Story 15.2). `Document` and `DocumentDetail` already carry `document_kind`, `needs_date_confirmation`, `partial_measured_at_text`.
- Backend `app/health_data/router.py:30-35` has `GET /health-values` with no query params. `app/health_data/repository.py::list_values_by_user` queries `HealthValue` alone — no JOIN to `documents`.

- `app/documents/service.py::delete_document` deletes health values and the document row; the `AiMemory` cascade handles AI cleanup. No backend change needed.

  - Ownership: every new endpoint MUST derive user identity from `Depends(get_current_user)`. Never accept `user_id` in a request body.
  - Preserve the existing 409/503 error shape for AI endpoints so the frontend's 409-vs-other discrimination doesn't need a separate branch.
- Frontend:
  - Svelte 5 runes only. `$state` for the filter store, `$derived` for anything computed from it, `$effect` only for side effects (localStorage writes). Do NOT introduce Svelte 4 stores.
  - TanStack Query: query keys MUST include the filter value so the cache distinguishes filters. Use prefix invalidation (`exact: false` or `predicate: q => q.queryKey[0] === 'ai_dashboard_interpretation'`) to sweep every filter variant on mutation.
  - 98.css idiom: the filter control should look and feel like the rest of the dashboard chrome. Prefer a 98.css `<fieldset>` + radio buttons over a `<select>` so the three options are always visible — matches the Windows 98 clinical workstation aesthetic.
  - Do NOT touch CSS bundles outside this feature's scope.

### Filter Semantics

- `all` → analysis + document (never unknown).
- `analysis` → only documents with extracted health values (`document_kind='analysis'` from Story 15.2).
- `document` → only documents that processed successfully with no extractable lab values (`document_kind='document'`).
- `unknown` is never exposed to the dashboard user. It represents a failure/unreadable classification and pollutes dashboard trust if included. Keep it out of every filter surface.
- Default filter is `'analysis'` because the product's primary use case is lab analysis. Users who want the broader view flip to `all` explicitly.

### Aggregate Prompt (dashboard interpretation)

The per-document prompt template (`INTERPRETATION_PROMPT_TEMPLATE`) is value-centric and doesn't generalize to an aggregate view. Introduce a new template `DASHBOARD_INTERPRETATION_PROMPT_TEMPLATE` that:

- States the scope explicitly: "You are summarizing the user's {filter_label} documents as of {iso_date}. Below are the AI interpretations already generated for each contributing document."
- Lists each contributing document's interpretation prefixed by its date (use `updated_at.date().isoformat()` already in `list_user_ai_context` output rows — do NOT expose document ids in the user-facing text).
- Asks for a plain-language, non-diagnostic overview of the dataset — NOT a re-interpretation of raw values.
- Enforces the same safety rules (no diagnosis, no medication, no treatment).
- Keep total response under 400 words (same budget as the per-document template).

Reuse `call_model_text` / `stream_model_text`, `validate_no_diagnostic`, `surface_uncertainty`, `inject_disclaimer`. Do NOT introduce a second safety pipeline.

`surface_uncertainty` currently takes `(text, values: list[NormalizedHealthValue])` — for the dashboard flow there are no raw `NormalizedHealthValue`s. Either pass an empty list (safe — the uncertainty flags are derived from per-value data that doesn't exist at aggregate scope) or add an overload. Prefer the empty-list path; the aggregate prompt already quotes per-document interpretations that went through the per-document pipeline, so the dataset-level uncertainty check is weaker by design.

### Endpoint Contract (for frontend alignment)

- `GET /api/v1/ai/dashboard/interpretation?document_kind=analysis` → `200 { document_id: null, interpretation: "…", model_version: "…", generated_at: "…", reasoning: null, source_document_ids: ["uuid1","uuid2"] }`
- `POST /api/v1/ai/dashboard/chat` with `{ document_kind: "all", question: "…" }` → `200 text/plain; charset=utf-8` streaming response; 409 `{ detail: "No analyses available for the active filter" }` when empty; 503 when model unavailable.
- `document_kind=unknown` is rejected by Pydantic Literal validation → 422.

### Rebuild Semantics (detailed)

- The dashboard aggregate is a pure function of `{all safety-validated AiMemory rows for user_id where Document.document_kind matches filter}`. There is no cached artifact on the backend.
- Frontend cache (`['ai_dashboard_interpretation', kind]` TanStack query) IS a cache. Invalidate it on every mutation that changes the underlying set:
  - `deleteDocument` success
  - Year-confirm success (`confirmDateYear`)
  - Any upload SSE event that transitions a document to `completed` or `partial` (already hooked for `['health_values']`)
- Do NOT manually invoke a "rebuild" endpoint — there is none. Rebuild == cache eviction + next GET.
- The `AiMemory` cascade delete on document delete is already wired. Confirm by running the existing 6-2 / 14-2 document-delete tests and verifying no new regression.
