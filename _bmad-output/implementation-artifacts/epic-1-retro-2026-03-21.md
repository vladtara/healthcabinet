## What Went Well

### 1. Infrastructure Baseline Was Exceptionally Solid
Story 1.1 delivered a complete, production-grade monorepo baseline — Docker Compose, K8s manifests, CI/CD pipelines, encryption layer, Alembic migrations, structlog, Ruff, mypy — all in a single story. Downstream stories built on this without revisiting fundamentals.

### 2. Architectural Discipline Held Across All 31 Review Rounds
The core boundaries never drifted: encryption locked to `repository.py`, business logic in service layer, routing only in router. Ruff enforced layer separation automatically. This is genuinely rare across a 4-story epic.

### 3. Test Coverage Was Proactive, Not Reactive
114 tests written by developers before QA had to request them. Story 1.3 alone shipped 57 tests covering rate limiting, refresh concurrency, inactivity timers, and cookie security attributes. Test coverage was treated as a first-class deliverable.

### 4. Team Caught Real Security Issues
- Timing-attack mitigation: `_DUMMY_HASH` / `_get_dummy_hash()` for user-not-found path
- Atomic Lua script for Redis rate limiting (eliminated INCR/EXPIRE race condition)
- PyJWT migration away from passlib/bcrypt v5 incompatibility (active CVE risk)
- PII suppression in exception messages and exception chain suppression

### 5. Forward-Looking Seeding for Epic 2
- K8s ingress SSE timeout annotation (120s) seeded in Story 1.1
- Conditional `Content-Type` header pattern preserved in Story 1.3 for multipart file uploads
- User tier system in Story 1.4 ready for upload rate limiting in Story 2.1

## What Didn't Go Well

### 1. Story 1.3 Required 12 Review Rounds
The login/session story had the highest complexity and the most review iterations. The same category of issues that appeared in earlier stories (layer violations, RFC 7807 shape, cookie attribute assertions) had to be re-caught rather than prevented.

### 2. Recurring Review Themes Across All Stories
The following issues appeared in 3 or more stories and should have been encoded as pre-review checks rather than discovered during review:
- **Layer separation violations** (router calling DB, service handling HTTP concerns) — flagged in Stories 1.1, 1.2, 1.3
- **RFC 7807 error shape drift** (missing `title`, `type`, `instance` fields) — flagged in Stories 1.1, 1.2, 1.3
- **PII in exception messages** (email in exception strings, unguarded exception chains) — flagged in Stories 1.2, 1.3

### 3. Unplanned Dependency Migration Mid-Story
PyJWT migration was not anticipated in Story 1.3 planning. Correct call, but signals dependency health audit should be part of epic prep, not mid-story firefighting.

### 4. Pre-existing `npm run check` Failure Carried Forward
Story 1.4 documented a broken `svelte-kit sync` caused by `.test.ts` files in auth route directories. This was accepted and deferred rather than fixed. It erodes toolchain confidence.

### 5. Per-Step Data Loss in Onboarding
Story 1.4 deferred per-step data persistence — data entered in step 1 or 2 is lost if the final submit fails. Documented as enhancement but affects user experience.

---

## Key Insights & Lessons Learned

1. **The architecture is solid; the process is expensive.** The design decisions are good. The missing piece is process gates (pre-review checklists) that catch known drift patterns before review begins. This would have cut Story 1.3 from 12 rounds to ~3-4.

2. **Recurring review themes = unwritten standards.** RFC 7807 shape, PII suppression, layer separation are de facto standards. They need to be written down and checked by developers before requesting review, not discovered by reviewers.

3. **Epic 2 is well-seeded.** SSE timeout annotation (1.1), conditional Content-Type (1.3), user tier system (1.4) — the scaffolding for document upload is already in place. This epic was built with forward intent that should be acknowledged and replicated.

4. **Technical debt is documented and bounded.** 7 debt items, all accepted with clear rationale and known impact. This is healthy. Undocumented debt is the real risk; this team has none.

5. **First-epic complexity is higher than subsequent epics.** Epic 1 established patterns, tooling, and conventions for the entire project. The elevated review rounds in Stories 1.2-1.4 partly reflect that pattern-discovery cost. Epic 2 should benefit from these established norms.
